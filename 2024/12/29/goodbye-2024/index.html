Goodbye 2024 Contest Reflection

This is my first contest reflection blog. In the future, I plan to write a blog and reflect on my mistakes and my successes after every contest. This not only helps with improvement, but it acts as a reminder to avoid past errors.

As a fun fact, this is my first time participating in and Div 1 + Div 2 combined contest.

TL;DR

  • + 11 rating (1802 -> 1813)
  • Solved ABCD
  • Rank 1882: 1831 performance
  • Wasted 40 mins misreading D

Problem A (492/500)

After reading the statement a few times, I observed that a range yields a solution if and only if twice the smallest is greater than the largest value. I spent several minutes thinking how to generalise this to full, before realising that I needed to find whether there exists at least two methods.

Looking at the sample, I realised that each value independently yields a solution. This motivated the approach to loop through all ranges greater than 1. All values outside the range can be taken individually.

This took roughly 3 minutes to implement, alongside 5 minutes of thinking of the solution.

Problem B (968/1000)

I quickly observed that if a position covers a range greater than 1, it will never interfere with anything else, since it can switch to a value that is different. This motivated the thought that some positions “lock” a value, and cannot be used for any others.

To answer a query, it was sufficient to check whether all values are locked inside the range it covers. This can be efficiently computed using prefix sums.

I came up with the idea in 2 minutes, with another 4 minutes to implement and debug my solution.

Problem C (1177/1250)

With only two input, but up to 2e9, this problem looked like a scary mathforces problem. I did not see any patterns or clear mathematical formulas to derive the answer when trying small cases. However, I did notice that whenever the range was split, each smaller box was symmetric. Furthermore, points where the sum was added was always a fixed distance away.

From this, I thought of a divide and conquer approach, to only calculate the left half, using the left half to derive the sum of the right half. I spent several minutes figuring out the finer details of this, but eventually managed to do it. My recursive function took in the size of the range, returning a pair. The pair stores (sum of range, number of elements taken).

With a 6 minute implementation and another 4 minutes of thinking, I solved problem C.

Problem D (1336/1750)

I struggled on D the most, and solving a misread version of D did not help.

For the first 40 minutes, I misread the problem into: maximise the product of max(a[i], b[i]) for all i

Fortunately, many parts of my misread were compatible with the actual solution. Since any permutaiton of B is valid, one can sort both A and B and compare.

The second observation related to maintaining the sorted nature of A and B (preferably without using PBDS). Since values increment by one each time, it suffices to find the last occurence and bump the value up by one, as it will never overtake an entire chunk.

Instead of dividing the modulo inverse and updating with the new value, I got lazy and implemented a segtree which stores the product.

Unfortunately, I implemented a segment tree that covers both arrays A and B, but forgot to remove it. I ended up with the segment tree supporting point query and point update operations.

This bumped the time complexity to O(n log^2 n), which TLE’d on pretest 6. A simple change into arrays reducted the time complexity to O(n log n), which passed easily.

Not only did that mistake cost 50 points, spending 40 minutes solving a more difficult variant cost a lot of points. It only took 20 minutes after I corrected my reasoning, which would have cost a lot of rating.

Problem E (0/2000)

I read the problem several times, but did not understand what I was supposed to find. I wasn’t too inclined to spend a long time on it, even though I had over 90 minutes remaining. Being weaker at game theory, I always strugged with this type of problem. Regardless, I should focus more time patching up my weaker sides. It took me 2.5 hours to solve USACO DEC24 Silver P1.

It was nearly 12:30 local time, and I decided to prioritise sleep over working on E.
